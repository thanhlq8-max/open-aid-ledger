# Donor Trust Guide

This guide explains how a future donor should evaluate Open Aid Ledger before sending support.

Current status remains inactive:

```text
DONATIONS_ACTIVE: NO
WALLETS_PUBLISHED: NO
ACTIVATION_APPROVED: NO
```

Do not send funds based on this document. This guide is a future-facing trust checklist and does not publish live receiving details.

## Plain-language promise

A donor should never need to trust a private message, a screenshot, or a single maintainer claim.

Before any future donation is accepted, the public repository should show:

- current activation status;
- approved campaign or support category;
- approved receiving channel inventory;
- exact asset and network;
- public verification method;
- ledger and report policy;
- account protection checklist;
- emergency freeze process;
- beneficiary privacy controls.

## What donors should check first

Before sending any support in a future active release, donors should verify:

- [ ] `DONATIONS_ACTIVE` is `YES` in the current public status.
- [ ] `WALLETS_PUBLISHED` is `YES` in the current public status.
- [ ] The receiving channel appears in an approved repository file.
- [ ] The asset and network match the donor's sending method.
- [ ] The public page links to the ledger and reporting policy.
- [ ] The account protection checklist has been reviewed.
- [ ] The campaign or support category is active.
- [ ] The public page does not promise yield, profit, repayment, or special access.

If any item is unclear, do not send.

## Simple future donor flow

When the project is ready, a donor-facing page should use this sequence:

```text
1. Check status
2. Choose approved campaign or support category
3. Choose asset and network
4. Copy receiving detail from the official public page only
5. Send from a compatible wallet or exchange
6. Save the transaction reference
7. Check public ledger/report after reconciliation
```

The project should not ask donors to trust direct messages or unreviewed screenshots.

## What good public information looks like

A safe future public page should contain:

| Item | Requirement |
|---|---|
| Status | clearly says active or inactive |
| Asset | exact symbol, for example `USDC` or `BTC` |
| Network | exact network name |
| Receiving detail | only from the approved repository source |
| Explorer | public verification link if applicable |
| Campaign | active campaign or support category |
| Risk note | no recovery guarantee for wrong network or wrong destination |
| Ledger | where the transaction will be recorded after reconciliation |
| Privacy note | no exposed personal details of supported people |
| Emergency note | what happens if the channel is frozen |

## Exchange users

A future donor may use an exchange if the exchange supports the selected asset and network.

The project should not require a donor to use a specific exchange. The donor must confirm network compatibility before sending.

Avoid wording that implies exchange support is guaranteed. Exchange policies can change.

## Red flags for donors

Do not send if you see:

- a receiving detail only in a private message;
- a mismatched asset or network;
- a request to send quickly before the public page is updated;
- a promise of profit, yield, repayment, or special benefit;
- a claim that support is active while public status still says inactive;
- an address or account not listed in an approved repository source;
- a request to share private account credentials;
- pressure to bypass the public ledger or report.

## What the maintainer must provide before activation

Before asking donors to send support, maintainers should provide:

- public status page;
- approved receiving channel inventory;
- account protection checklist completion;
- campaign or support category status;
- ledger publication policy;
- reconciliation schedule;
- incident freeze procedure;
- beneficiary privacy rules;
- legal and tax review notes where applicable.

## Why donations are inactive now

The inactive status protects donors, maintainers, and supported people while the project hardens governance, reporting, account protection, legal review, and privacy controls.

Inactive status is not a failure. It is a safety gate.
